Date: 2026-05-18

- `docs/plan/map-spec.md`

## Goal

The map is the primary game surface. It must render reliably on mobile, show the player location, display nearby game objects, and support the loop of finding pitches, recruiting players, and acting on proximity.

The engineering direction is to follow GeoBlazor and ArcGIS component APIs as much as possible. RuckR code should own game rules, data loading, product overlays, and diagnostics. It should not grow into a framework around GeoBlazor.

## GeoBlazor Surface

`GameMap.razor` owns a GeoBlazor `MapView` with:

- `Map > Basemap > BasemapStyle` for the ArcGIS Navigation basemap when `Map:BasemapMode=styled`.
- No basemap when `Map:BasemapMode=empty`, for reduction/debug testing.
- `GraphicsLayer` for player location.
- `GraphicsLayer` for encounters.
- `GraphicsLayer` for candidate places.
- `GraphicsLayer` for pitches.
- GeoBlazor widgets: `HomeWidget`, `CompassWidget`, `LocateWidget`, `ScaleBarWidget`, and `LegendWidget`.
- `OnViewInitialized` for lightweight view readiness and non-layer startup work.
- `OnViewRendered` for the first graphics-layer sync.

GeoBlazor layer declaration order matters: the first declared layer renders above later layers. The current order is intentional:

1. `Player location`
2. `Encounters`
3. `Candidate places`
4. `Pitches`

This keeps player location and recruitable encounters visually above pitch markers.

## Runtime Flow

1. Player opens `/map`.
2. `MapView` initializes and the page marks the map as available.
3. Authentication state and game progress load.
4. `OnViewRendered` fires after GeoBlazor has rendered the view and attached child map layers.
5. The first graphics sync runs once:
   - Player location graphic is added if a last known GPS position exists.
   - Nearby pitches load.
   - Candidate places load when enabled.
   - Encounters load when authenticated.
6. Browser GPS watch updates `LocationState`.
7. Movement updates player location and refreshes nearby game objects when distance/time thresholds are met.
8. Marker taps use GeoBlazor `MapView.HitTest` and RuckR selection attributes to open RuckR overlays.

## Remaining Work

### 1. Verify Native Legend Output

The GeoBlazor `LegendWidget` is now wired in. Because pitch, encounter, candidate, and player location graphics use per-graphic symbols inside `GraphicsLayer`s, the native legend may not show category-level rows the same way a renderer-backed layer would.

Next checks:

- Deploy current commits.
- Use mobile and desktop smoke checks to inspect the legend content.
- Check diagnostics for `nativeLegendVisible`.
- If the legend is too shallow, evaluate a renderer-backed layer for pitches.

Acceptance:

- The legend does not block map interaction on Pixel.
- The legend gives useful map-symbol context, or the limitation is documented with a renderer-backed follow-up.

### 2. Evaluate Renderer-Backed Pitch Layer

The pitch layer is the best remaining candidate for deeper GeoBlazor/ArcGIS alignment. A renderer-backed layer could move pitch-type styling out of per-graphic symbol construction and improve native legend behavior.

Questions:

- Does GeoBlazor Core 4.4.4 expose the layer type and renderer APIs needed for an in-memory pitch feature layer?
- Can we preserve selection, attributes, and offline/client-owned update behavior?
- Does this reduce `GameMap`/`MapGraphicFactory` complexity enough to justify the migration?

Acceptance:

- Keep the current `GraphicsLayer` if renderer-backed data requires more custom plumbing.
- Migrate only if the layer API is clearly more idiomatic and improves legend/symbol behavior.

### 3. Keep Startup Lifecycle Aligned With GeoBlazor

Current policy:

- `OnViewInitialized`: readiness, authentication, game progress, diagnostics.
- `OnViewRendered`: first layer sync.
- Position changes after render: update location and refresh nearby objects.

Acceptance:

- No layer mutation depends on child `GraphicsLayer` refs before `OnViewRendered`.
- No repeated full reload on every render cycle.
- Map loading overlay clears promptly even if services settle slowly.

### 4. Expand Automated Coverage

Manual device testing is reserved for user-reported or device-specific issues. The stable paths should be automated.

Add or maintain coverage for:

- `/map?basemap=styled&mapGraphics=true&autoGps=false`
- `/map?basemap=styled&mapGraphics=false&autoGps=true`
- `/map?basemap=styled&mapGraphics=true&autoGps=true`
- Maintain pitch marker tap coverage so it opens the RuckR overlay without native popup duplication.
- Maintain player location layer coverage for mocked GPS dot and accuracy circle graphics.
- Maintain native map widget and native legend coverage when widgets are enabled.

Current targeted command:

```powershell
dotnet test RuckR.Tests\RuckR.Tests.csproj --filter "FullyQualifiedName~RuckR.Tests.E2E.MapReductionTests"
```

### 5. Retire Temporary Diagnostics When Stable

Diagnostics have been useful for Pixel and deployment debugging. They should stay lightweight but not become permanent UI architecture.

Keep:

- Client console forwarding to server telemetry.
- Map health summary for production troubleshooting.

Reduce later:

- Verbose DOM/canvas probes.
- Debug route exposure in nav.
- Old Pixel-specific troubleshooting notes once automated tests cover the behavior.

## Definition Of Done

- Pixel and desktop render the full game map reliably.
- Map layers follow GeoBlazor documented ordering.
- Initial layer sync follows GeoBlazor lifecycle and waits for rendered view state.
- Native widgets and legend are preferred over custom map controls where practical.
- RuckR custom code is limited to game behavior, product overlays, data loading, diagnostics, and small adapters.
- Automated tests cover stable map behavior; manual checks are reserved for user-reported regressions.
