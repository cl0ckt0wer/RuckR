Scope

The review treated /api/map/encounters as a player-facing, state-mutating gameplay endpoint. The adversarial lens focused on broken invariants, replay/idempotency, privilege and data isolation, race behavior, location spoofing edges, null/time boundaries, and test determinism.

Findings

MediumEncounter creation remains non-atomic, so concurrent refreshes can create duplicate active encounters for the same user and player.

References: RuckR/Server/Services/RecruitmentService.cs:93-104, RuckR/Server/Services/RecruitmentService.cs:212-244, RuckR/Server/Data/RuckRDbContext.cs:97-100.

Exploit or failure scenario: an authenticated user opens two tabs or scripts parallel calls to /api/map/encounters. Both requests can sample the same player, both can enter GetOrCreateEncounterAsync, both can observe no active row for that user/player, and both can insert because the database only has a non-unique index on UserId, PlayerId. The new stability guarantee assumes a single active encounter row per user/player, but that invariant is not enforced at the database or transaction boundary.

Evidence: GetOrCreateEncounterAsync performs a read-before-insert check and then calls SaveChangesAsync without a transaction, lock, or unique constraint. SQL Server's default read committed isolation does not prevent the classic duplicate insert race. The index configured in RuckRDbContext is not unique.

Suggested fix: enforce the invariant in the database and service. The simplest durable model is one row per UserId, PlayerId, replacing expired or moved rows in place; back it with a unique index and handle duplicate insert exceptions as a retry/read path. If multiple historical rows must be retained, move active state to a separate table or serialize by user/player in a transaction.

MediumThe new active encounter hydration query is unbounded per user before the visible cap is applied.

References: RuckR/Server/Services/RecruitmentService.cs:52-64, RuckR/Server/Controllers/MapController.cs:32-49.

Exploit or failure scenario: a logged-in user can sweep map coordinates inside the allowed radius range and cause GET requests to create active encounters. Later refreshes load every unexpired encounter for that user, include each player, and filter in memory against the current parks before taking only 8 visible rows. With today's seed count this is bounded by the available player table, but the path scales with total active rows, not with the visible board size.

Evidence: GetEncountersAsync fetches all active, uncollected encounters for the user via ToListAsync(), then applies nearbyParks.Any(...), ordering, and Take(MaxVisibleEncounters) in memory. MapController.GetEncounters rate-limits radius but does not rate-limit the state-changing GET or bind it to the server's last known GPS position.

Suggested fix: make the active encounter lookup selective before materialization. Options include storing the source park/place id on PlayerEncounterModel and filtering by current nearby park ids, adding a coarse latitude/longitude bounding box before loading, or applying an indexed per-user active cap. Also consider applying the existing rate-limit service to map encounter generation because this GET mutates state.

LowThe new regression test verifies only sequential refresh stability and does not cover duplicate, overflow, or concurrency cases.

References: RuckR.Tests/Api/RecruitmentApiTests.cs:123-145.

Exploit or failure scenario: the test can pass while the board contains duplicate players after concurrent refreshes, while more than 8 active records exist in storage, or while older active records outside the current park set are repeatedly relocated. Those are the adversarial cases most likely to violate the player-facing stability expectation.

Evidence: the test issues two immediate sequential GET requests against the default test park and asserts first encounter ids are present in the second response. It does not assert unique PlayerId values, storage-level uniqueness, active row count, behavior when active visible encounters exceed 8, or parallel request behavior.

Suggested fix: add targeted coverage for no duplicate active UserId, PlayerId rows after rapid refresh, board size never exceeding 8, stable selection when more than 8 active nearby encounters exist, and collected players being removed from the active board.

Residual Risk

  • /api/map/encounters remains a state-mutating GET. That may be acceptable for gameplay discovery, but it should be treated as write-like for idempotency, replay, and rate limiting.
  • The review did not run full solution tests or production smoke tests because the requested scope was the local working tree diff and focused tests were practical.
  • The review did not inspect all client map rendering paths. It assumed the documented server invariant is the primary change under review.
  • No source fix was applied. The findings are recommendations for the owning implementation pass.

Researched Methodology

This review used a manual, threat-driven code review frame rather than a broad checklist. The method was based on:

  • OWASP Code Review Guide, which keeps manual review prominent for issues automated tools miss.
  • OWASP Secure Code Review Cheat Sheet, which frames secure review as manual examination of code for vulnerabilities and supports baseline plus incremental review.
  • OWASP Threat Modeling Process, used here to prioritize high-risk code paths by actor, asset, trust boundary, and abuse scenario.

Applied adversarial questions: can an authenticated user make the endpoint do more work than intended, duplicate state by replaying or racing requests, bypass location assumptions, see or mutate another user's data, keep expired or collected records visible, or make the regression test pass while the product invariant fails?
